슬랙에 스크린샷 하나가 올라왔습니다. "봄 원피스"로 검색했는데, 첫 줄에 뜬 게 두꺼운 패딩 사진이었죠.

그 아래는 더 이상했습니다. 강아지랑 같이 찍은 컷인데 무드엔 `러블리`, 카테고리엔 `아우터`가 붙어 있었어요. 그런데 사진 속 사람은 반팔 차림이었습니다.

아우터를 입은 건… 강아지였고요.

다들 웃었습니다. 저도 웃었죠. 그리고 3초 뒤, 등골이 서늘해졌습니다. 이게 과연 하나였을까요?

착장 사진에서 색상, 핏, 카테고리, 무드를 자동으로 뽑는 기능. 데모로는 반나절 만에 만들었습니다. 멀티모달 모델에 사진을 던지고 스타일을 설명해달라고 하면 그럴듯한 답이 돌아왔거든요.

그 반나절짜리 데모가 프로덕션에서 3주짜리 사고 수습으로 바뀌는 과정. 그 이야기를 해보려 합니다.

## 강아지 사건은 시작에 불과했다

로그를 뒤지기 시작했습니다. 웃긴 게 하나 나오면, 그 뒤엔 항상 열 개가 더 있는 법이죠.

커피를 든 채 찍은 사진은 종이컵을 `액세서리`로 태깅했습니다. 전신 거울 셀카는 거울에 비친 뒷배경 옷걸이까지 착장으로 인식했고요. 어두운 실내에서 찍은 반팔 사진은, 배경이 칙칙하다는 이유로 시즌이 `겨울`로 찍혔습니다.

하나같이 사람이라면 하지 않을 실수였어요.

그중 제일 아찔했던 건 이겁니다. 로고가 전혀 없는 무지 후드티에, 모델이 브랜드를 붙여놓은 케이스.

사진 어디에도 없는 브랜드 이름을, 그냥 "이 정도 핏이면 이 브랜드겠지" 하는 느낌으로 지어낸 거였습니다. 큐레이션이나 상품 연결이 걸린 서비스에서 브랜드를 잘못 붙이면, 그건 웃고 넘길 일이 아니죠. 엉뚱한 상품으로 링크가 걸리고, 그 뒤로 줄줄이 틀어집니다.

정리하면 문제는 세 갈래였습니다.

**모델은 형식을 안 지켰습니다.** 어떤 응답은 문장으로, 어떤 응답은 목록으로, 어떤 응답은 "이 사진은 감각적인 무드가 돋보이는..." 하는 사족부터 붙였어요. 파서가 이 변주를 못 따라가고 계속 깨졌습니다.

**모델은 빈칸을 못 견뎠습니다.** 상반신만 나온 사진인데도 하의를 기어이 `슬림핏 데님`으로 채웠어요. 모르면 모른다고 하면 될 걸, 확률 모델은 그럴듯한 값을 지어내는 쪽을 택합니다.

**모델은 같은 질문에 다르게 답했습니다.** 똑같은 사진을 두 번 넣으면 한 번은 `캐주얼`, 한 번은 `스트릿`이 나왔어요. 헷갈릴 만한 경계이긴 해도, DB에 저장될 값이 호출할 때마다 바뀐다는 건 곤란했습니다.

로그를 보다가 이 출력을 발견하고서야, 접근 자체가 틀렸다는 걸 받아들였습니다.

```json
{
  "color": "네이비 계열의 차분한",
  "fit": "약간 루즈한 느낌",
  "category": "아우터(코트류로 추정됨)",
  "mood": "미니멀하면서도 캐주얼한",
  "season": "가을~겨울"
}
```

색상 칸에 "차분한"이 들어 있고, 카테고리엔 "추정됨"이라는 주석이 달려 있고, 무드는 두 개가 섞여 있습니다.

이걸 그대로 검색 인덱스에 넣으면 `color=네이비 계열의 차분한` 같은, 아무도 검색하지 못하는 데이터가 쌓입니다. 프롬프트를 아무리 갈아도 이건 근본적으로 안 사라졌어요. 조금 덜해질 뿐이었죠.

## 나는 모델을 잘못 대하고 있었다

한동안 프롬프트만 붙잡고 있었습니다. "브랜드가 안 보이면 지어내지 마" 같은 문장을 계속 추가했죠. 조금 나아지면 다른 데서 터졌습니다. 두더지 잡기였어요.

관점이 바뀐 건 이 문장을 떠올리고 나서였습니다.

**LLM은 응답을 보장하지 않는, 신뢰할 수 없는 외부 시스템이다.**

결제 API를 붙일 때를 생각해보면 답이 나옵니다. 우리는 외부 API 응답이 늘 정상일 거라고 믿지 않죠. 타임아웃을 걸고, 스키마를 검증하고, 실패하면 재시도하거나 폴백합니다. 응답 본문을 곧이곧대로 DB에 넣는 개발자는 없어요.

LLM도 똑같이 대했어야 했습니다. 다만 여기서 못 믿을 건 서버 가용성이 아니라 **출력의 형식과 내용**이라는 점이 달랐을 뿐이죠.

프롬프트로 모델을 완벽하게 길들이겠다는 생각을 버리자, 할 일이 명확해졌습니다. 불완전한 출력을 전제로 깔고, 그 출력이 시스템 안으로 들어오는 경계를 설계하는 것.

## 강아지를 막은 세 개의 문

### 첫째, 값을 마음대로 못 쓰게 막는다

가장 먼저 자유 텍스트를 없앴습니다. 색상을 자유롭게 적게 두니 "네이비 계열의 차분한"이 나오잖아요. 그러면 애초에 고를 수 있는 값을 정해진 목록으로 닫아버리면 됩니다.

Kotlin에선 enum이 딱이죠.

```kotlin
enum class Category { TOP, BOTTOM, OUTER, DRESS, SHOES, ACCESSORY }
enum class Fit { SLIM, REGULAR, LOOSE, OVERSIZED }
enum class Mood { MINIMAL, CASUAL, STREET, FORMAL, VINTAGE }

data class StyleAttributes(
    val category: Category,
    val fit: Fit,
    val mood: Mood,
    val colors: List<String>,   // 사전 정의한 팔레트 안에서만
    val season: Season,
    val confidence: Double,     // 모델이 스스로 매긴 확신도
)
```

그리고 모델에는 자유 서술 대신, 이 스키마에 맞춘 구조화 출력(structured output)을 강제했습니다. 요즘 멀티모달 API는 대부분 JSON 스키마를 주면 그 형태로만 답하게 제약할 수 있어요.

이것만으로 "형식이 매번 다르다" 문제의 상당 부분이 사라졌습니다. 이제 모델에겐 문장을 쓸 자유가 없으니까요.

브랜드처럼 지어내기 쉬운 값은 한술 더 떴습니다. 사진에서 로고 텍스트가 실제로 읽히지 않으면, 아예 `null`로 두게 했어요. 못 믿을 값은 받지 않는 게 낫습니다.

### 둘째, 검증하기 전엔 안 믿는다

스키마를 강제해도 끝이 아니었습니다. 구조화 출력은 형식만 맞춰줄 뿐, 내용이 말이 되는지는 보장하지 않거든요. 강아지 아우터도, 거울 속 옷걸이도 형식은 완벽했습니다. 값이 틀렸을 뿐이죠.

그래서 모델과 저장소 사이에 검증 계층을 하나 세웠습니다. 팔레트 밖 색상, enum 밖 값, 논리적으로 안 맞는 조합을 여기서 걸러냅니다.

```kotlin
@Service
class StyleAttributeValidator(
    private val colorPalette: Set<String>,
) {
    fun validate(raw: RawExtraction): ValidationResult {
        val errors = buildList {
            if (raw.colors.any { it !in colorPalette }) {
                add("팔레트에 없는 색상: ${raw.colors - colorPalette}")
            }
            if (raw.brand != null && !raw.logoTextDetected) {
                add("로고가 안 읽혔는데 브랜드가 채워짐 → 지어낸 값 의심")
            }
            // 상반신 사진에 하의 속성이 붙는 등의 모순 검사
        }
        return if (errors.isEmpty()) ValidationResult.Valid(raw.toAttributes())
        else ValidationResult.Invalid(errors)
    }
}
```

원칙은 하나입니다. 검증을 통과 못 한 출력은 데이터로 치지 않는다.

애매하면 버립니다. 태깅 안 된 사진 한 장이, 잘못 태깅돼서 검색 결과를 오염시키는 사진 한 장보다 나으니까요. 빈 값이 틀린 값보다 낫습니다.

### 셋째, 모델이 헷갈리면 사람에게 넘긴다

모델에 확신도를 같이 뱉게 한 데는 이유가 있습니다. 캐주얼과 스트릿 사이처럼 모델도 헷갈리는 사진은, 억지로 자동 분류하지 않는 편이 낫거든요. 이런 건 사람 검수 큐로 보냅니다.

```kotlin
when (val result = validator.validate(raw)) {
    is ValidationResult.Invalid -> reviewQueue.enqueue(ugcId, result.errors)
    is ValidationResult.Valid -> {
        if (result.attributes.confidence < CONFIDENCE_THRESHOLD) {
            reviewQueue.enqueue(ugcId, "확신도 미달")
        } else {
            attributeRepository.save(ugcId, result.attributes)
        }
    }
}
```

전부 자동화하려다 데이터 품질을 망치느니, 확실한 것만 자동으로 처리하고 나머지는 사람이 보는 편이 결과적으로 빨랐습니다. 자동화율의 목표는 100%가 아니라, "믿을 수 있는 만큼"이어야 했어요.

## 그리고 청구서가 날아왔다

경계를 다 세우고 한숨 돌릴 때쯤, 이번엔 비용이 발목을 잡았습니다.

멀티모달 호출은 텍스트보다 비싸고, 이미지 토큰은 해상도에 비례해 늘어납니다. 하루 수만 장을 원본 그대로 밀어 넣으니 청구서가 감당이 안 됐어요.

두 가지로 손봤습니다.

하나는 속성 추출을 실시간 경로에서 빼는 것. 유저가 업로드하는 그 순간에 동기로 모델을 부를 이유가 없습니다. 업로드 이벤트를 받아 비동기로 돌리면 되고, 사용자 경험엔 지장이 없죠.

```kotlin
@Async
@EventListener
fun onUgcUploaded(event: UgcUploadedEvent) {
    val resized = imageProcessor.resize(event.imageUrl, maxWidth = 768)
    val raw = styleExtractor.extract(resized)
    handleExtraction(event.ugcId, raw)
}
```

다른 하나는 모델에 넣기 전 이미지를 줄이는 것. 스타일 속성을 판별하는 데 원본 해상도는 필요 없습니다. 긴 변을 적당히 리사이즈하면 이미지 토큰이 눈에 띄게 줄고, 정확도는 거의 그대로예요. 리사이즈 한 줄이 추론 단가를 크게 낮췄습니다.

## 강아지가 남긴 것

그 스크린샷 하나가 결국 파이프라인 전체를 다시 짜게 만들었습니다.

지나고 보니 고마운 사고였어요. 데모만 보고 넘어갔다면, 강아지 아우터도 지어낸 브랜드도 조용히 데이터에 쌓였을 테니까요.

착장 사진에서 스타일을 뽑는 문제는, 프롬프트를 잘 쓰는 문제처럼 보입니다. 막상 시간을 쏟은 곳은 따로 있었어요. 모델 출력을 어디까지 믿을지, 어디서 끊을지, 못 믿을 값을 어떻게 처리할지를 정하는 경계. 정작 프롬프트가 차지한 비중은 생각보다 작았습니다.

그리고 이건 착장 태깅에만 해당하는 이야기가 아닙니다. LLM을 백엔드 파이프라인 어딘가에 끼워 넣는 모든 자리에 똑같이 적용돼요.

모델을 신뢰할 수 없는 외부 시스템으로 놓는다. 스키마로 값을 닫는다. 검증한 뒤에만 믿는다. 애매하면 사람에게 넘긴다.

강아지가 가르쳐준 네 줄입니다.

다음 글에서는 이렇게 뽑아낸 속성이 실제 검색과 추천에서 어떻게 쓰이는지, 자연어 스타일 검색을 어떻게 붙였는지를 이어서 다루겠습니다.
